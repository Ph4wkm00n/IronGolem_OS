# UX Mission and Pillars

## Mission

Make a powerful autonomous platform feel simple, safe, and understandable for
every user - from non-technical operators to security-conscious admins.

## Core UX Promise

Users should feel:
- **Informed**, not overwhelmed
- **Protected**, not restricted
- **Assisted**, not replaced
- **In control**, even when the system acts in the background

## Experience Pillars

### 1. Visible Trust
- Show permissions clearly before actions execute
- Show risk before action, not after
- Show what was blocked and why
- Show what was healed and how recovery worked

### 2. Managed Complexity
- **Basic users** see recipes and summaries
- **Advanced users** can inspect squads, traces, and policies
- **Admins** can manage tenants, channels, and incidents
- Advanced features are always available but never required

### 3. Ambient Operations
- Heartbeats, recoveries, and research updates should feel like calm system
  maintenance, not noisy alerts
- Suppress-on-OK: only surface information when it matters
- Status indicators convey health without demanding attention

### 4. Explainable Autonomy
- Every automation needs a human-readable contract
- Every squad needs a plain-language role explanation
- Every blocked or quarantined event needs a simple cause statement
- "Why did this happen?" must always be one click away

## Writing Style

- Use plain language on all default surfaces
- Avoid jargon: "orchestrator," "vector," "tenant," "OTLP"
- Replace technical labels with outcome-oriented labels:
  - "Assistant team" instead of "agent squad" in beginner mode
  - "Workspace" instead of "tenant" on default screens
  - "Safety rules" instead of "policy engine" on onboarding screens
